# 类 Instagram App 开发规范文档（uni-app + Vue3）

# 前言

本规范基于 uni-app + Vue3 技术栈，规范开发流程、编码标准、目录结构及第三方使用，确保代码可维护、可扩展，适配 iOS 上架要求。

# 一、基础规范

## 1.1 开发环境规范

- Node.js：使用 18.x 或 20.x 版本
- 开发工具：HBuilderX 或 VS Code（需安装 uni-app、Vue3 插件）
- 包管理：统一使用 npm，依赖版本固定（禁止使用 ^、~ 符号）
- 编译环境：使用 vite 编译，测试与生产环境严格区分

## 1.2 版本控制规范

- 分支管理：
  - main：生产环境分支，仅用于发布，禁止直接提交
  - develop：开发主分支，用于测试环境部署
  - feature/xxx：功能分支，如 feature/post-publish
  - bugfix/xxx：bug 修复分支，如 bugfix/video-play-error

- 提交规范：格式为”类型: 描述”
  - feat：新增功能
  - fix：修复 bug
  - docs：文档更新
  - style：代码格式调整
  - refactor：重构代码
  - 示例：feat: 新增帖子滤镜功能

# 二、目录结构规范

目录结构遵循”功能模块化、职责单一”原则，禁止随意新增一级目录。

```plain text

说明：

- 目录、文件命名使用”小写横线分隔”，如 post-detail.vue
- components 目录中，组件按功能分类，每个组件单独创建文件夹
- pages 目录中，每个页面模块单独创建文件夹
- 静态资源优先放在 assets 目录，无需编译的资源放在 static 目录

## 2.1 接口文档管理规范（重要）

在 `assets/api-docs/` 目录下，按模块创建接口文档，每个接口必须包含以下信息：

**文档格式示例（user.md）：**

```markdown
# 用户模块接口文档

## 1. 用户登录

- 接口名称：用户登录
- 请求路径：/api/user/login
- 请求方式：POST
- 请求参数：
  ```json
  {
    “username”: “string”,  // 用户名
    “password”: “string”   // 密码
  }
  ```
- 返回值：
  ```json
  {
    “code”: 200,
    “message”: “登录成功”,
    “data”: {
      “token”: “string”,
      “userInfo”: {
        “id”: “string”,
        “username”: “string”,
        “avatar”: “string”
      }
    }
  }
  ```

## 2. 获取用户信息

- 接口名称：获取用户信息
- 请求路径：/api/user/info
- 请求方式：GET
- 请求参数：
  ```json
  {
    “userId”: “string”  // 用户ID
  }
  ```
- 返回值：
  ```json
  {
    “code”: 200,
    “message”: “success”,
    “data”: {
      “id”: “string”,
      “username”: “string”,
      “avatar”: “string”,
      “followCount”: 0,
      “fansCount”: 0
    }
  }
  ```
```

**规范要求：**
- 每个模块的接口文档独立维护
- 接口变更时必须同步更新文档
- 新增接口必须先写文档再开发
- 文档格式统一，便于查阅

# 三、编码规范

## 3.1 Vue3 编码规范

- 组件声明：使用 setup 语法糖（`<script setup>`）
- 组件命名：Pascal 命名，如 PostCard、ImageCrop
- Props 定义：使用 defineProps，必填项标注 required
- Emits 定义：使用 defineEmits，事件名使用 kebab-case
- 响应式数据：ref 用于基本类型，reactive 用于对象类型
- hooks 封装：重复逻辑封装为 composables，命名以 use 开头

**模板规范：**
- 缩进 2 个空格
- v-bind 简写为 :，v-on 简写为 @
- v-for 必须指定唯一 key，禁止使用 index
- 复杂逻辑封装为 computed 或方法

**样式规范：**
- 使用 scss，选择器使用”小写横线分隔”
- 组件样式使用 scoped 修饰
- 主题色、边距、圆角等使用变量，禁止硬编码
- 优先使用 rpx 单位，使用 flex 布局

## 3.2 接口请求规范

- 接口封装：所有接口在 api 目录下封装，按模块划分文件
- 请求配置：全局配置在 api/index.js 中统一配置
- 请求拦截器：统一添加 Token、设备信息
- 响应拦截器：统一处理错误（Token 过期、接口报错）
- 请求方法：遵循 RESTful 规范（GET 查询、POST 提交、PUT 更新、DELETE 删除）
- 错误处理：失败时统一提示，特殊错误在拦截器中处理
- 加载状态：请求时显示 loading，完成后关闭

## 3.3 Pinia 状态管理规范

- 状态划分：按功能模块划分（userStore、postStore、appStore）
- 状态命名：小写驼峰，如 userInfo、postList
- 状态操作：禁止直接修改 state，必须通过 action 修改
- 持久化：需要持久化的状态使用 pinia-plugin-persistedstate 插件

# 四、第三方插件使用规范

- 版本控制：使用指定版本，禁止随意升级
- 插件选择：优先使用 uni-app 官方插件、成熟稳定的插件
- 插件封装：封装为工具函数或组件，禁止直接调用插件 API
- iOS 审核：禁止使用存在审核风险的插件（热更新、违规权限）

# 五、功能开发规范

## 5.1 通用功能规范

- 加载状态：数据请求时显示 loading，完成后显示内容，失败时显示错误提示
- 空状态：列表无数据时显示空状态图片+提示文字
- 错误处理：所有操作添加错误处理，避免崩溃
- 交互体验：
  - 按钮点击需有视觉反馈
  - 列表支持下拉刷新、上拉加载
  - 图片支持手势缩放、滑动切换
  - 视频支持自动静音播放、暂停/继续、全屏
- 兼容性：适配 iOS 12.0 及以上版本

## 5.2 核心功能开发规范

**发帖功能：**
- 支持多图选择（最多 9 张）
- 图片正方形裁剪（1:1 比例）
- 发布前验证内容，显示进度条
- 未发布的帖子自动保存至草稿箱

**帖子展示：**
- 首页使用瀑布流布局
- 双击图片点赞，显示爱心动画
- 视频自动静音播放

**互动功能：**
- 点赞/取消点赞实时更新状态
- 评论支持回复，实时更新列表
- 关注成功后推送通知

**消息功能：**
- 区分不同通知类型（点赞、评论、关注、系统）
- 未读消息显示红点提示

# 六、iOS 上架规范

- 权限配置：在 manifest.json 中正确配置权限，添加权限申请说明
- 隐私政策：必须包含隐私政策页面，首次打开需弹出同意弹窗
- 内容审核：帖子、评论、头像等需有审核机制
- 热更新：禁止用于修改核心功能
- 注销账号：必须提供注销功能
- IPv6 支持：确保支持 IPv6 网络

# 七、代码审核规范

- 提交前自检：确保符合规范，无语法错误、无多余注释
- MR 审核：至少 1 名成员审核，审核通过后才可合并
- 注释规范：
  - 函数、组件需添加注释说明功能、参数、返回值
  - 复杂逻辑添加行内注释
  - 禁止无用注释，注释需与代码同步更新
